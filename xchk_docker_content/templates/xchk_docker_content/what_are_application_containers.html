Schriftelijk materiaal

Zelfs als je (nog) geen IT-professional bent, ben je waarschijnlijk al conflicten tussen applicaties tegengekomen. Je hebt bijvoorbeeld een stukje software A geïnstalleerd en deze software werkte perfect tot je andere software B installeerde. Dat kan bijvoorbeeld zijn omdat A en B dezelfde systeemresources gebruiken. Voor de gewone consument is dit vaak gewoon vervelend. Voor professionele gebruikers, bijvoorbeeld softwarebedrijven, is dit gevaarlijk en duur.

Een simpele, maar onrealistische oplossing: installeer A op één machine en installeer B op een andere machine. Dan kan er geen conflict meer zijn. In de praktijk is dit natuurlijk onbetaalbaar. Bovendien moeten al deze machines onderhouden worden.

Een flexibelere oplossing bestaat erin virtuele machines te gebruiken. Hierbij installeer je een of meerdere gastbesturingssystemen binnenin je gewone hostbesturingssysteem. Voor elke gast wordt er een stukje capaciteit van je systeem gereserveerd. Anders gezegd, als je een sterke processor hebt, kan je bijvoorbeeld meerdere gastsystemen installeren die werken alsof ze toegang hebben tot een middelmatige processor. Op die manier heb je meerdere virtuele machines, terwijl je maar één fysieke machine hebt.

Een tweede oplossing bestaat erin containers te gebruiken. Containers bevatten geen eigen besturingssysteem. Dit levert bepaalde voor- en nadelen ten opzichte van virtuele machines. Het belangrijkste voordeel is dat ze veel "lichter" zijn. Een besturingssysteem is een enorm stuk software. Het belangrijkste nadeel is dat ze minder flexibel zijn. Als je een container uitvoert op een zuiver Linuxsysteem, kan je geen Windowsapplicaties installeren in deze container.

Als je al wat ervaring hebt met containers, weet je misschien dat je wel Linuxapplicaties in Docker containers kan installeren op Windows. Zonder in veel detail te gaan, is dat omdat dat geen zuivere Windowssystemen zijn. Er zijn bepaalde aanpassingen gedaan om dit mogelijk te maken.
